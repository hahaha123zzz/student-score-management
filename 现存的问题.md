# 现存问题与改进方向

## 功能限制

1. **成绩修改历史缺失**：当前成绩录入后会直接覆盖旧成绩（`grades.csv` 中更新对应行），无法追溯修改记录。建议后续加入 `grade_history` 表或 `logs.txt` 成绩变更日志记录每次修改前后的分数差异。
2. **并发处理能力弱**：服务器采用 `while(true) { accept → recv → process → send → close }` 单线程同步阻塞模型，每次只能处理一个客户端请求。当多个浏览器同时访问时，后到的请求必须排队等待，高并发场景下用户体验较差。可考虑引入多线程（`std::thread` 每连接一线程）或 select/poll 多路复用。
3. **无数据备份与恢复机制**：所有数据以 CSV 文件存储在 `data/` 目录下，无自动备份或快照功能。若误删除文件或程序异常退出导致 CSV 写入不完整，数据将永久丢失。建议加入定时自动备份（如每天备份到 `data/backup/`）或手动导出全部数据的功能。
4. ~~**考试锁定不可逆**~~（已修复）：已添加 `PUT /api/exams/{id}/unlock` 接口，解锁后状态恢复为草稿
5. ~~**科目不可删除**~~（已修复）：现已支持 `PUT /api/subjects/{id}` 和 `DELETE /api/subjects/{id}`
6. **端口号硬编码**：服务端监听端口固定为 8080（`server.cpp:239` 中 `htons(8080)`），无法通过命令行参数或配置文件修改。若端口被占用，用户需修改源码并重新编译。

## 前端表现问题

1. **管理后台表格对齐偏差**：`admin.html` 中成绩管理页面的表格列数量会随考试科目动态变化。当通过 JS 动态渲染 `<th>` 和 `<td>` 时，若列数变化后未同步更新 `colspan` 或 `table-layout` 样式，可能导致表头与数据列错位。
2. **未做响应式设计**：前端页面（`login.html`、`admin.html`、`student.html`）未使用 CSS 媒体查询或弹性布局，在手机等小屏设备上显示时会出现横向滚动条、按钮重叠、表格溢出等问题。
3. **单页应用路由缺失**：管理后台所有功能模块（学生管理、成绩管理、统计分析等）集中在单一 `admin.html` 页面中通过 Tab 切换显示，但 URL 不会变化。浏览器前进/后退按钮无法在模块间导航，刷新页面后回到默认模块而非之前操作的模块。
4. **加载状态反馈不足**：部分 API 请求（如 CSV 批量导入、统计分析）响应时间较长时，页面无 loading 动画或进度提示，用户可能误以为页面卡死而重复点击。

## 已知 Bug

1. **Windows 控制台中文乱码（低版本系统）**：Windows 7 及以下系统默认使用 GBK 编码的控制台，若未调用 `SetConsoleOutputCP(CP_UTF8)`，`std::cout` 输出的中文提示信息会显示为乱码。Windows 10 及以上版本默认支持 UTF-8，不受此影响。
2. **Token 会话在服务器重启后失效**：Token 认证信息存储在 `utils_base.cpp` 中的 `static std::map<std::string, std::string> tokenToUserId` 内存映射表中，程序重启后所有已登录用户的 Token 全部清空，需重新登录。代码中已明确注释"这是最简单的 token 管理方案，适合学习，不适合生产环境"（`utils_base.cpp:133`）。
3. **rand() 随机数可预测**：Token 生成使用 `rand()` 而非密码学安全的随机数生成器（`utils_base.cpp:217-240`）。`srand(time(NULL))` 以秒级时间戳播种，攻击者若能推算服务器启动时间，即可预测所有生成的 Token。
4. **JSON 手动拼接存在注入风险**：所有 API 响应通过手动字符串拼接构建 JSON（如 `handlers_grade.cpp:31-50`），若数据字段中包含未转义的双引号或反斜杠，可能破坏 JSON 结构导致前端解析失败。`jsonEscape` 函数提供了基础转义但调用不全面。
5. **CSV 解析不支持引号包裹字段**：`storage.cpp` 中的 `splitCSV` 按逗号简单分割，若学生姓名中包含逗号（如 "张,三"），会被错误切分为两个字段。标准 CSV 格式应用双引号包裹含逗号的字段，当前不支持。
6. ~~**考试状态无"撤回中"校验**~~（已修复）：`retractExam` 现仅允许撤回 "published" 状态考试；locked 考试需先解锁再撤回

## 安全相关限制

1. **密码哈希强度不足**：密码通过 DJB2 哈希算法存储（`utils_base.cpp:105-122`），该算法设计用于哈希表索引而非密码学安全。无盐值（salt），相同密码产生相同哈希值，无法抵御彩虹表攻击。建议改用 SHA-256 或 bcrypt。
2. **无请求频率限制**：API 接口（包括登录）无限流机制，攻击者可暴力枚举密码试错。登录接口无失败次数锁定或验证码保护。
3. **CORS 配置过于宽松**：`Access-Control-Allow-Origin: *` 允许任意来源跨域访问（`server.cpp:388`），任意网站的 JavaScript 均可调用本系统 API。若服务暴露在公网，存在 CSRF 攻击风险。
4. **无 HTTPS 支持**：服务器仅处理 HTTP 明文协议，用户密码和 Token 在网络中明文传输，局域网内可被嗅探截获。
5. **权限控制不完善**：`handlers_auth.cpp:47-51` 中 `isAdmin` 函数仅区分 admin 与非 admin，教师账号（T001-T003）和 admin 账号共享同一 admin 角色。教师无法仅管理自己班级的学生和成绩。

## 跨平台说明

本项目已通过条件编译（`#ifdef _WIN32` / `#elif defined(__APPLE__)` / `#else`）兼容 Windows、macOS、Linux 三平台，但存在以下平台差异：

- **可执行文件名**：Windows 为 `edugrade.exe`，macOS/Linux 为 `edugrade`。
- **编译工具链**：Windows 需安装 Visual Studio（MSVC）或 MinGW；macOS/Linux 可使用系统自带 g++/clang。MSVC 编译需指定 `/utf-8` 标志确保源码中的中文注释正常编译。
- **Socket 库**：Windows 使用 WinSock2（需链接 `ws2_32.lib`，需 `WSAStartup`/`WSACleanup` 初始化与清理）；macOS/Linux 使用 POSIX Socket（无需额外初始化）。
- **路径分隔符**：`PATH_SEPARATOR` 宏在 Windows 定义为 `"\\"`，macOS/Linux 为 `"/"`。`server.cpp:154-158` 的 `readStaticFile` 函数在拼接文件路径时已处理此差异。
- **文件系统 API**：目录创建（`_mkdir` vs `mkdir`）、文件存在检查（`_access` vs `access`）、本地时间获取（`localtime_s` vs `localtime_r`）在不同平台使用不同函数。
- **可执行路径获取**：Windows 用 `GetModuleFileNameA`，macOS 用 `_NSGetExecutablePath`，Linux 读 `/proc/self/exe` 符号链接（`main.cpp:14-33`）。
- **编码支持**：Windows 需要系统版本 ≥8（Windows 10）才能良好支持 UTF-8 控制台输出；macOS/Linux 终端默认 UTF-8。

## 代码架构改进建议

1. **路由分发方式**：当前 `main.cpp` 使用大量 `if-else` 分支匹配 URL 路径分发请求，每新增一个 API 需手动添加分支。建议使用 `std::map<std::string, HandlerFunc>` 路由表注册模式，减少 `main.cpp` 的维护负担。
2. **JSON 构建方式**：全项目通过手动字符串拼接构建 JSON，代码冗长且易出错。建议引入轻量级 JSON 库（如 nlohmann/json header-only 库），或至少封装 `jsonBuildObject()` / `jsonBuildArray()` 辅助函数。
3. **错误处理统一化**：部分 handler 函数内部使用 `return utils::errorResponse("...")`，而另一部分通过 try-catch 或无错误返回，缺乏统一的错误处理和日志记录规范。
4. **数据层抽象**：当前 handler 直接调用 `storage::getXxx()` / `storage::saveXxx()`，若后续从 CSV 切换到 SQLite 等数据库，需逐文件修改。建议引入数据访问层（DAO）接口，将 CSV 操作封装在实现类中。
5. **单元测试缺失**：项目无任何自动化测试（如 Google Test 或 Catch2），所有功能仅靠手动浏览器测试验证。建议为核心模块（storage、stats、sort、models）补充单元测试。
